# Xuan-Flow 记忆系统多维优化需求文档（V1）

## 1. 文档目标

本文档用于定义 Xuan-Flow 下一阶段的记忆系统升级需求，作为后续架构设计、开发实现和验收测试的统一依据。

本次升级的核心方向：
- 保留现有 JSON 原子化记忆作为第一层（会话原子层）。
- 新增 memory.md 作为第二层（实时工作记忆层，仅放当前任务有用记忆）。
- 引入 MySQL 作为第三层（长期记忆层，保存完整记忆集合）。
- 建立可扩展的多维度记忆优化机制（写入、召回、冲突消解、可观测性）。

## 2. 背景与现状

当前系统已经具备以下能力：
- 对话结束后通过 LLM 抽取原子事实（fact）。
- 事实包含 category、confidence、createdAt、updatedAt、source 等字段。
- 使用本地 JSON 文件进行存储，并支持规范化、去重、置信度筛选。
- 在系统提示词注入阶段，根据置信度和时间进行有限条目召回。

当前主要限制：
- 文件存储对并发写入、跨实例部署、审计追踪支持不足。
- 记忆召回维度较少（主要是 confidence + 时间），缺少业务语义和任务上下文维度。
- 缺少长期记忆生命周期管理（失效、归档、软删除、版本追踪）。
- 缺少可量化评估体系，难以持续优化记忆质量。

## 3. 升级目标（业务目标）

将记忆能力从“会存”升级为“可持续决策支持”：
- 在复杂多轮任务中提高上下文连续性和个性化稳定性。
- 在不明显增加上下文 token 的前提下，提高记忆命中率与回答一致性。
- 支持多会话、多用户、多实例下的稳定持久化与检索。

目标结果：
- 用户在新会话中能明显感知到系统“记得住且记得准”。
- 研发侧可追踪每条记忆的来源、变化、命中和失效原因。

## 4. 范围定义

### 4.1 本期范围（In Scope）
- 三级存储架构：
  - L1：JSON 原子化记忆（会话级原子层、低延迟读写）。
  - L2：memory.md 工作记忆（任务级上下文层，仅注入当前有用记忆）。
  - L3：MySQL 长期记忆（结构化、可查询、可审计，保存完整集合）。
- 记忆写入策略优化：
  - 置信度阈值。
  - 去重与合并策略。
  - 冲突覆盖与版本留痕。
- 记忆召回策略优化：
  - 任务相关性。
  - 最近性。
  - 置信度。
  - 业务类别优先级。
- LangGraph 编排对接：
  - 会话内先构建并注入 memory.md。
  - 对话后异步写回长期记忆层（MySQL），并刷新 L1/L2。

### 4.2 非本期范围（Out of Scope）
- 向量数据库引入（可作为下一期扩展）。
- 跨语言实体对齐与知识图谱构建。
- 复杂权限系统（仅保留基础租户/用户隔离字段）。

## 5. 核心设计原则

- 分层解耦：编排层、抽取层、存储层、召回层独立演进。
- 原子优先：最小语义单元存储，避免整段对话堆积。
- 可解释：每次写入、召回、淘汰都可追踪原因。
- 可回滚：任何自动更新都保留变更轨迹与版本记录。
- 可降级：MySQL 异常时可回退到 L1 + L2，保证主链路可用。

## 6. 目标架构（三级记忆）

### 6.1 L1：JSON 原子化记忆（会话原子层）
定位：
- 作为会话级快速原子事实池。
- 优势是低复杂度、低依赖、读写快。

职责：
- 保存会话内最新抽取得到的原子事实（fact）。
- 提供构建 L2（memory.md）的原始候选数据。
- 在 L3 异常时作为近期记忆兜底来源。

### 6.2 L2：memory.md 工作记忆（实时层）
定位：
- 作为当前任务的“工作记忆面板”，仅包含对当前需求有用的记忆。
- 由系统在每轮请求前动态生成，可快速替换。

职责：
- 从 L1 + L3 召回候选记忆并重排。
- 只保留 Top-K 有用记忆，控制注入 token 成本。
- 直接作为系统提示词记忆上下文输入。

### 6.3 L3：MySQL 长期记忆（持久层）
定位：
- 作为长期事实的权威存储。
- 支持结构化查询、审计、生命周期管理。

职责：
- 持久化多用户多会话完整记忆（重要与不重要均保留）。
- 支持按维度检索和统计评估。
- 支持记忆版本、冲突记录、软删除和归档。

### 6.4 读写路径
- 写入路径：对话结束 -> LLM 抽取事实 -> 写入 L1（原子事实） -> 同步落库 L3（完整集合） -> 依据当前任务生成/刷新 L2（memory.md）。
- 召回路径：任务开始 -> 从 L3（必要时补 L1）按多维排序召回 -> 构建 L2（仅有用记忆） -> 注入 LangGraph 上下文 -> 生成响应。

## 7. 多维度优化模型

### 7.1 写入维度
- 置信度维度：低于高价值阈值不进入 L2，但仍可保留在 L3（可标记 low_value）。
- 稳定性维度：重复出现的事实提升稳定分。
- 来源维度：用户明确陈述 > 推断信息。
- 冲突维度：新旧事实冲突时保留版本并标注生效状态。

### 7.2 召回维度
- 相关性维度：与当前任务关键词/意图的匹配程度。
- 时间维度：最近更新优先，但不绝对覆盖高稳定事实。
- 置信度维度：高置信事实优先。
- 类别维度：goal/preference/context 的动态优先级。
- 使用频次维度：高命中且有效的事实优先。
- 输出约束：L2 仅保留本轮有用记忆，避免全量注入。

### 7.3 生命周期维度
- 衰减机制：长期未命中且低稳定分记忆降权。
- 失效机制：被明确否定或过期事实标记 inactive。
- 归档机制：低频但重要事实归档保存，按需召回。

### 7.4 质量维度
- Precision：被召回记忆中真正有帮助的比例。
- Recall：关键记忆在任务中被召回的比例。
- Conflict Rate：冲突事实占比。
- Hallucination-From-Memory：由错误记忆导致答复偏差的比率。

## 8. MySQL 数据模型（建议）

### 8.1 表设计（V1）
- memory_fact
  - id (PK)
  - tenant_id
  - user_id
  - thread_id
  - fact_key（规范化内容哈希）
  - content
  - category
  - confidence
  - stability_score
  - status（active/inactive/archived）
  - source_type（explicit/inferred/tool）
  - created_at
  - updated_at
  - expired_at（可空）

- memory_fact_version
  - id (PK)
  - fact_id
  - previous_content
  - new_content
  - previous_confidence
  - new_confidence
  - change_reason
  - changed_at

- memory_retrieval_log
  - id (PK)
  - tenant_id
  - user_id
  - thread_id
  - task_id
  - retrieved_fact_ids（可 JSON）
  - retrieval_reason（可 JSON）
  - created_at

### 8.2 索引建议
- memory_fact: (tenant_id, user_id, status, category)
- memory_fact: (tenant_id, user_id, updated_at desc)
- memory_fact: unique(tenant_id, user_id, fact_key)
- retrieval_log: (tenant_id, user_id, created_at desc)

## 9. LangGraph 集成要求

- ThreadState 中保留会话态字段，不将长期事实全量放入 state。
- 记忆召回在进入模型前执行，生成 L2（memory.md）后注入。
- 记忆写入在响应后异步触发，先更新 L1，再落 L3，最后刷新 L2。
- 工具调用链路需记录“哪些记忆参与了本轮决策”，用于回溯。

## 10. 功能性需求（FR）

- FR-1：系统必须支持 JSON + memory.md + MySQL 三级记忆读写。
- FR-2：系统必须支持记忆事实去重、合并、冲突标注。
- FR-3：系统必须支持按多维分值排序召回记忆。
- FR-4：系统必须支持记忆失效与软删除。
- FR-5：系统必须记录记忆召回日志和版本变更日志。
- FR-6：系统必须支持“L2 仅保留本轮有用记忆”的动态构建策略。

## 11. 非功能性需求（NFR）

- NFR-1：主对话响应延迟不能因记忆写入显著增加（写入异步化）。
- NFR-2：召回接口在常规规模下需满足低延迟查询（可通过索引保障）。
- NFR-3：MySQL 不可用时系统可降级使用 L1 + L2，核心对话不应中断。
- NFR-4：所有记忆写入与更新必须可审计。

## 12. 迁移方案

### 阶段 A：抽象存储接口
- 从当前 memory store 抽离统一存储接口（MemoryRepository）。
- 保持 JSON 实现不变作为 baseline，并定义 L2 构建接口（WorkingMemoryBuilder）。

### 阶段 B：接入 MySQL 实现
- 增加 MySQL Repository。
- 支持配置开关写入模式（json_only/triple_write）。

### 阶段 C：召回策略升级
- 在读取链路引入多维排序函数，并落地 L2 生成模板。
- 增加命中日志与效果统计。

### 阶段 D：治理能力
- 引入失效、归档、版本回溯。
- 补充管理端查询接口与观测指标。

## 13. 验收标准

- 能在配置中启用三级架构，并控制 L2（memory.md）动态生成。
- 在相同测试集下，记忆召回准确性有可量化提升。
- 发生冲突更新时可查询到版本变更记录。
- MySQL 故障演练下，系统可自动降级并保持主链路可用。

## 14. 风险与应对

- 风险：抽取错误导致脏记忆累积。
  - 应对：提高写入阈值、增加冲突检测、支持人工纠偏。

- 风险：多维排序过于复杂导致维护成本高。
  - 应对：先实现线性加权模型，再逐步升级学习排序。

- 风险：双写一致性问题。
  - 应对：以 L3 为权威源，L1/L2 周期性从 L3 重建。
